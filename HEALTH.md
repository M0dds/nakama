## Offen

Volle Begründung je Finding steht in der Session die es aufdeckte (`git log` rund um den HEALTH.md-Initialcommit).

- **A5** — `episodesQueryOptions` macht 4-5 Round-Trips pro Item-Page-Load. **Paging-Umbau erledigt** (Thema 2, Branch `feat/paging`: `limit`-Wachstum → `.range()`-Fenster, identische Round-Trip-Zahl). Die **Reduktion selbst bleibt offen**: der einzige echte Hebel wäre, das doppelte Item-Resolve (episodesQueryOptions löst `(type,slug)→id` auf, obwohl ItemDetails `item`-Query das schon hat) zu sparen — das würde die beiden Queries aber **serialisieren** (episodes wartet auf item.data) statt sie parallel laufen zu lassen → kein klarer Wall-Clock-Gewinn. Zurückgestellt, bis ein Symptom auftaucht.
- **AUD-2** — *(aus Audit 2026-06-01, bewusst NICHT gefixt)* Migration `20260531100000` droppt `unique(user_id, episode_id)` und ersetzt ihn durch zwei partielle Indizes. Logbook-Ära-RPCs mit `on conflict (user_id, episode_id) do nothing` brauchen aber einen nicht-partiellen Arbiter → würden `42P10` werfen. **Nakama selbst verschont** (nacktes `on conflict do nothing`). Übersprungen, weil **Logbook nicht mehr genutzt wird** (User-Entscheidung). Falls Logbook je reaktiviert wird auf derselben DB: betroffene RPCs auf nacktes `on conflict do nothing` umstellen.
- **CAP-1 (teil-gefixt 2026-06-01)** — Supabase erzwingt einen **harten 1000-Zeilen-Cap (`db-max-rows`)**, der jedes größere `.limit()` überstimmt (die alte „`.limit(5000)` reicht"-Annahme war falsch). **Gefixt** (Whole-Show-Queries, die real >1000 zogen): Mitseher-Auge (`coWatchersOptions`, per-Seiten-Scope, `41a1d3b`) + Episodentitel-Backfill (`selectTitleGaps`, `.range()`-Paginierung + `TITLE_ENRICHMENT_VERSION` 4→5). **Noch offen, aber praktisch unkritisch** (fensterbasiert, bei Paar/Freundeskreis-Skala < 1000): `findItemsWithNewEpisodes` (lists.ts, list_items-Aggregat → Badge-Undercount ab 1000 list_items) + Kalender-Fenster (`calendar.ts`, ~6-Monats-Airings). Echte Lösung dort: `.range()`-Chunking, falls je ein Power-User drauf läuft. Kommentare an beiden Stellen entsprechend korrigiert.
- **PRIV-1 (gefixt `20260601110000`)** — `episode_watches_select_co` globale Lane band an `is_co_member(episode_id, viewer)` (prüft nur Item-Zugang des Betrachters, nicht den Watcher) → **echtes Cross-User-Lese-Leck** (jeder sah fremde globale Watch-Historie). Fix: `shares_list_with(user_id, auth.uid())`. **Voll-Audit danach:** alle SELECT-Policies + DEFINER-RPCs mit Daten-Rückgabe → sauber. Siehe handshake §Gotchas → Daten/RLS.
- **AUD-10 · PRELAUNCH-1/2/3 (gefixt `20260601130000`)** — siehe Bundle 9. PRELAUNCH-3 nur halb: `item_history`-Redundanz beseitigt, **`profiles`-Doppel-Policy bewusst belassen** (das `_co_member` hat keine Self-Klausel → `profiles_select_own` ist die einzige Self-Read-Garantie für listenlose User, NICHT redundant — Audit-Fehldiagnose).
- **SEC-DEPLOY (Security-Audit 2026-06-04) — Code erledigt, Deploy-Schritte offen (→ Phase 9).** `steam-proxy` ist gehärtet: per-User-Auth via `supabase.auth.getUser()` (401 ohne echte Session — der Client sendet jetzt das Session-`access_token` als Bearer statt des Anon-Keys, steam.ts), CORS auf `ALLOWED_ORIGINS`-Allowlist statt `*` gepinnt, Param-Allowlist je Endpoint. **Beim Deploy noch zu tun:** (1) `npx supabase secrets set ALLOWED_ORIGINS="https://<prod-origin>"` (sonst blockt CORS alles — safe default), (2) `npx supabase functions deploy steam-proxy`. **Optional/deferred:** Rate-Limit/Caching (Kosten-Schutz) — für Paar/Freundeskreis-Skala unkritisch. Kein aktives Risiko (Function nicht deployed). Details: `SECURITY-AUDIT.md` L-2/I-3.
- **SEC-BASELINE (aus Security-Audit 2026-06-04) — Sicherheits-Layer erledigt, Tabellen-DDL bewusst offen.** Der komplette Live-**Sicherheits-Layer** (alle SECURITY-DEFINER-Funktionen + RPCs, RLS-Enablement, alle Policies, Trigger) ist jetzt in `supabase/migrations/00000000000000_baseline.sql` snapshottet (per SQL-Editor-Introspektion gezogen, da `supabase db dump` Docker braucht). Details: `SECURITY-AUDIT.md` I-1.
- **SEC-TMDB (aus Security-Audit 2026-06-04, low, bewusst)** — `VITE_TMDB_TOKEN` ist ins Client-Bundle inlined (read-only v4-Token, geringer Blast-Radius — kein Account-/Write-Zugriff, keine PII; Risiko = Quota-Abuse).
